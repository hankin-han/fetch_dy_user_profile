## ✅ 最终方案：极简参数调 API + localStorage 缓存

**关键发现**：用户提供的代码能正常工作，核心区别是：
- 用 `aweme_id` key（不是 `item_id`）
- 极简参数：`aweme_id` + `cursor` + `count` + `device_platform=webapp` + `aid=6383`
- 不加任何额外的 browser/screen/os 等参数
- 不加 `a_bogus`/`msToken`/`webid` 等签名
- `credentials: 'include'` 带 cookie + `Referer` header

**改动**：
- 删除所有逆向代码（getPageTokens, generateABogus, buildCommentUrl, clearTokenCache，约300行）
- 新增 `buildApiUrl()` 极简 URL 构建
- 新增 localStorage 缓存（loadFromLocalStorage/saveToLocalStorage，key: dy_comment_cache_{aweme_id}）
- 加载优先级: localStorage → 内存 cache → 网络拉取
- 重新解析和 clearCache 同步清除 localStorage
**根因**：抖音评论 API 需要完整浏览器运行环境，纯 fetch 无法模拟。

**最终方案**：模拟真实用户浏览行为

| 组件 | 改动 |
|------|------|
| `dy_comment_module.js` | 全量重写。数据获取：locaStorage缓存 → window.open popup → postMessage监听。UI保留（弹窗/表格/排序/分页/导出） |
| `fetch_dy_user_profile.js` | 在文件最开头（旧实例清理之前）新增 ~140 行 popup 检测和评论提取代码 |

**工作流程**：
1. bz.hankin.cn 点评论数 → 查 localStorage 缓存
2. 无缓存 → window.open 小窗口到 douyin.com/video/{aweme_id}
3. douyin.com popup 窗口上，脚本检测 window.opener 存在 → 提取评论区数据（3层策略：script标签JSON → RENDER_DATA深层搜索 → DOM结构解析）
4. postMessage 发回 bz.hankin.cn
5. bz.hankin.cn 接收 → 存 localStorage → 渲染表格
6. 下次同作品秒开

**部署**：两个文件都需要上传到服务器

## 修复：评论数红色链接失效问题

**问题**：作品列表中的评论数不显示红色，也无法点击。

2. **修复：评论模块加载路径错误** — `script.src = './dy_comment_module.js'` 相对路径是相对于当前页面 URL 解析的（而非脚本所在目录），导致加载 404。改用 `document.currentScript.src` 获取主脚本真实 URL，再拼接同目录下的 `dy_comment_module.js` 绝对路径。
3. **alert 提示增加加载地址** — 点击评论数时若模块未加载，alert 会显示实际请求的 URL（`window.__DyCommentUrl__`），方便排查 404 问题。
4. **评论模块初始化异常兜底** — 整个 IIFE 包裹 try/catch，异常时输出详细错误（类型/消息/堆栈）。主文件 onload 回调验证 DyComment 是否存在，指向控制台错误。
5. **评论 API 请求增强** — fetchWithRetry 异常时输出完整响应（status_code/status_msg/keys）。请求参数增加浏览器指纹（screen/browser/platform 等），URL 拼接改为 `findScriptBase()` 兼容性方案。
6. **mixed-content 修复** — 脚本加载从硬编码 `http://` 改为 `document.currentScript.src` 自动匹配协议，避免 HTTPS 页面加载 HTTP 资源被屏蔽。

7. **评论 API 反爬签名** — `status_code=5` 是因为缺少 `msToken` + `a_bogus` 等反爬参数。新增：
   - `getPageTokens()` 从 cookie/localStorage/页面 script 标签提取 msToken/webid/verifyFp/uifid
   - `generateABogus()` X-Bogus 签名生成（CRC32 变体 + 编码表 + URL 参数排序）
   - `buildCommentUrl()` 统一构建带签名的 API URL
   - `clearTokenCache()` 重新解析时刷新 token
   - fetchCommentPage/fetchAllReplies 改为使用 buildCommentUrl

**根因**：`text-dy-red` 是自定义 Tailwind 颜色，当 HTML 通过 JavaScript 模板字符串动态注入时，Tailwind CDN 的 JIT 编译器无法可靠生成对应的 CSS 规则，导致颜色不生效，用户以为不可点击（实际事件委托正常）。

**修复内容**：
1. 在静态 CSS 块新增 `.dy-comment-link` 样式规则（含 `!important`），完全绕开 Tailwind CDN
2. span 标签增加 inline style 作为双重保障

## 修复：评论 API status_code=5（a_bogus 签名无效） — 三层备选策略

**问题**：自实现的 `generateABogus()` 签名算法与抖音真实 X-Bogus 不匹配，API 始终返回 status_code=5。

**方案**：放弃自实现签名，改用三层递进备选策略：

**策略1 — 简化 API（首选）**：
- `buildCommentUrl` 裁剪为最小参数集：`device_platform=webapp`, `aid=6383`, `channel=douyin`
- 与主页面调用 profile/post API 的模式完全一致（主页面也不需要 a_bogus）
- 去掉 msToken/webid/verifyFp/uifid/a_bogus 等所有反爬参数

**策略2 — HTML 嵌入提取（仅首页）**：
- 新增 `fetchCommentsFromPageHTML()` — fetch 视频页 HTML
- 从 `<script id="RENDER_DATA">` / `window.__INITIAL_STATE__` / `SSR_DATA` 提取嵌入评论
- 支持多路径兼容（新旧版本抖音）
- 局限：只能拿到第1页（~20条），无法翻页

**策略3 — PHP 服务器代理（终极备选）**：
- 新增 `proxy_douyin.php` — 接收浏览器 cookie，服务器端 curl 请求 Douyin
- 新增 `fetchViaProxy()` — 前端调用代理
- 路径白名单校验（只允许 `/aweme/v1/web/`）
- 需要部署到 `https://bz.hankin.cn/static/chrome/proxy_douyin.php`

**修改文件**：
- `dy_comment_module.js`：buildCommentUrl 简化、新增 fetchCommentsFromPageHTML/fetchViaProxy、fetchCommentPage 三层 fallback、fetchAllReplies 简化参数
- `proxy_douyin.php`：新增，PHP 代理脚本
